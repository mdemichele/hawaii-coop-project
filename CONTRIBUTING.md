# Contributing to Hawaii Co-op Project

Thank you for your interest in contributing. This project exists to make it easier for communities in Hawaii to form and operate cooperative organizations — housing co-ops, agricultural co-ops, and beyond. Every contribution, large or small, moves that mission forward.

## Code of Conduct

This project is built around the co-op values of democracy, equity, and community. We expect contributors to treat each other with the same respect. Be kind, be direct, and assume good intent. Harassment or discrimination of any kind will not be tolerated.

---

## Ways to Contribute

You don't have to write code to contribute meaningfully:

- **Report a bug** — Open an issue describing what you found and how to reproduce it.
- **Suggest a feature** — Open an issue describing the problem you're trying to solve and your proposed approach.
- **Improve documentation** — Fix typos, clarify unclear sections, or add missing context.
- **Review pull requests** — Thoughtful code review is one of the most valuable contributions you can make.
- **Write code** — Pick up an open issue and submit a pull request.
- **Share domain knowledge** — If you have experience with Hawaii co-op law, DCCA filings, HRS Chapter 421 or 421H, agricultural programs, or community organizing in Hawaii, your knowledge can shape how this platform works. Open an issue or start a discussion.

## Getting Started

1. **Fork the repository** and clone your fork locally.
2. **Read the docs.** The `docs/` directory contains the product overview, feature specifications, architecture, and individual feature design documents. Getting familiar with these will help you understand how the pieces fit together before writing any code.
3. **Check open issues** for anything labeled `good first issue` or `help wanted`.
4. **Open an issue before starting significant work.** If you're planning a non-trivial change, open an issue first to discuss the approach. This avoids duplicated effort and makes it much more likely your PR will be accepted.

## Branching & Pull Requests

- All changes must go through a pull request — direct pushes to `main` are not permitted.
- Branch off `main` and use a short, descriptive branch name:
  ```
  feature/membership-equity-ui
  fix/formation-vote-quorum-calculation
  docs/update-architecture
  ```
- Keep pull requests focused. One logical change per PR makes review easier and keeps the git history readable.
- Fill out the PR description with what changed and why. If your PR closes an issue, include `Closes #123` in the description.
- At least one approving review is required before a PR can be merged.

---

## Commit Messages

- Use the imperative mood for the subject line: `Add equity ledger table` not `Added equity ledger table`.
- Keep the subject line under 72 characters.
- If the change needs more explanation, add a blank line after the subject and write a short body paragraph.
- Reference related issues in the body when relevant: `Closes #42`.

---

## Reporting Issues

When opening a bug report, please include:

- A clear description of the problem
- Steps to reproduce it
- What you expected to happen
- What actually happened
- Any relevant environment details (browser, OS, etc.)

For feature requests, describe the problem you're trying to solve rather than jumping straight to a specific solution — it helps keep the discussion open to the best possible approach.
